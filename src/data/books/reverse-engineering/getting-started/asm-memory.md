# 字符串、内存与交叉引用

## 本章目标

前面几章学了寄存器、指令、栈帧——这些都是"看懂代码"的基础。但逆向时你面对的是一个陌生的程序，第一件事不是看懂每一行，而是**找到关键代码在哪里**。

这一章教你三个最实用的定位技能：

1. **搜索字符串** — 程序里的提示文字是最直接的线索
2. **内存窗口** — 直接看程序的数据长什么样
3. **交叉引用** — 追踪"谁用了这个数据/代码"

上一章第1章你已经用搜索字符串找到了 "Correct!" 并破解了程序。这章会更系统地讲这三种方法。

## 字符串：程序的"口供"

程序里的字符串是最直接的线索。错误提示、按钮文字、URL、注册码提示——这些文字直接告诉你程序在做什么。

### ASCII 字符串（窄字符串）

每个字符占 **1 字节**，末尾以一个 `00`（NULL 字节）结束。

```
"Hello" 的存储方式：
地址    字节
401040  48 65 6C 6C 6F 00
        H  e  l  l  o  \0
```

右侧 ASCII 列会直接显示 "Hello"。`00` 是字符串结束标志——程序读字符串时从起始地址开始一个字节一个字节读，遇到 `00` 就停。

### Unicode / UTF-16LE 字符串（宽字符串）

每个字符占 **2 字节**，末尾以 `00 00`（两个 NULL 字节）结束。Windows 内核原生使用这种格式。

```
"Hello" 的 UTF-16LE 存储方式：
地址    字节
401040  48 00 65 00 6C 00 6C 00 6F 00 00 00
        H     e     l     l     o     \0\0
```

每个字符后面都跟着一个 `00`，看起来像每个字母之间插了空格。

### 程序怎么"传"字符串

程序引用字符串时，**传的是字符串的起始地址**，不是内容本身：

```asm
push 0x00401040          ; "Hello" 的起始地址
call printf              ; printf 从这个地址开始读，遇到 00 停止
```

就像你告诉快递员"去某某地址取包裹"——你给的是地址，不是包裹本身。`printf` 拿到地址后自己去内存里读字符串内容。

在 x64dbg 里你看到 `push <..."Hello"...>` 这种指令，push 后面的值就是字符串的地址。双击它可以跳到字符串在内存中的位置。

<!-- 📸 截图：x64dbg 中 push 字符串地址的指令 -->

### 在内存窗口认字符串

打开内存窗口（点击下方的"内存"标签页），你会看到两列：

```
地址         十六进制                          ASCII
012FF300    48 65 6C 6C 6F 00 FF FF FF...    Hello....
012FF310    45 6E 74 65 72 20 70 61 73...    Enter pas
012FF320    73 77 6F 72 64 3A 00 00 00...    sword:...
```

右边 ASCII 列能直接看到 "Hello" 和 "Enter password:"。这是找线索最直观的方式。

### 为什么有时候找不到字符串？

有些程序会把字符串加密或混淆——存储时是乱码，运行时才解密。你在 x64dbg 里搜索不到明文字符串，就是因为这个。高级的逆向会需要先找到解密函数，把字符串解密后再分析。入门阶段遇到这种情况可以先跳过。

## x64dbg 搜索字符串（回顾 + 深入）

第1章你已经用这个方法找到了 "Correct!"。这里补充一些技巧：

### 基本操作

1. 在 CPU 窗口的反汇编区域**右键**
2. **Search for** → **Current Module** → **All referenced text strings**
3. 弹出字符串列表窗口

<!-- 📸 截图：字符串列表窗口 -->

### 字符串列表窗口技巧

- **双击**字符串 → 跳转到引用该字符串的代码
- 窗口上方有**搜索框**，可以输入关键词过滤（比如输入 "pass" 过滤出含 "pass" 的字符串）
- 列表里会同时显示 **ASCII** 和 **Unicode** 字符串，Unicode 的前面通常有 `L` 标记或显示为宽字符

### 搜索 vs 内存窗口的区别

- **搜索字符串**：找的是代码里**引用**了哪些字符串，直接跳到引用的代码位置
- **内存窗口**：直接看内存里存了什么数据，不管代码有没有引用

两个方法互补。搜索字符串定位代码，内存窗口验证数据。

## 内存窗口：看程序的数据

### 打开和导航

内存窗口在 x64dbg 下方，和 CPU 窗口、堆栈窗口并列。如果没有看到，点击菜单 **View** → **Memory**。

按 **Ctrl+G**（Go to）输入地址，可以跳转到任意内存位置。

### 看字符串数据

1. 用字符串搜索找到 "Enter password:" 双击跳转
2. 跳转到了 `push <..."Enter password:"...>` 这行指令
3. push 的值就是字符串地址（比如 `0x0040A010`）
4. 切换到内存窗口，按 Ctrl+G 输入这个地址
5. 看到十六进制和 ASCII 对照

<!-- 📸 截图：内存窗口中显示 "Enter password:" 字符串 -->

### 看栈上的数据

函数调用时参数和局部变量都在栈上。你可以直接看：

1. 在寄存器窗口找到 ESP 的值（比如 `0x012FF2E4`）
2. 切换到内存窗口，Ctrl+G 输入 ESP 的值
3. 看到栈上的数据——可能包含函数参数、局部变量、返回地址
4. 按 F7 单步执行 push 指令，观察内存窗口中新数据出现

<!-- 📸 截图：内存窗口中显示栈数据 -->

### 修改内存数据

你可以直接修改内存中的值来测试程序行为：

1. 在内存窗口找到要修改的位置
2. **右键** → **Binary** → **Edit**
3. 在弹出的对话框中输入新的十六进制值
4. 点确定

比如把 "reverse2026" 改成 "abcdefghij"（长度要一致），然后运行程序输入 "abcdefghij"，看看程序是否认为密码正确。

<!-- 📸 截图：修改内存中的字符串 -->

## 交叉引用：谁用了这个地址

**交叉引用（xref）** 记录的是"谁引用了这个数据或代码"。它是追踪程序逻辑的利器。

### 两种交叉引用

**代码交叉引用**（Code Xref）——谁跳转或调用到了这里：

```
; 这条指令被 0x00401020 处的 jne 引用了
00401030  push <..."Correct!"...>    ; ← CODE XREF: 00401020 j
```

意思是：地址 `0x00401020` 有一条 `jne` 指令可能跳到这里。双击 `00401020j` 可以跳回去看那条跳转指令。

**数据交叉引用**（Data Xref）——谁读取了这个数据：

```
; 这个字符串被 0x00401050 处的 push 引用了
0040A010  "Enter password:", 00      ; ← DATA XREF: 00401050 push
```

意思是：地址 `0x00401050` 有一条 `push` 指令引用了这个字符串。双击可以跳到使用它的代码。

### x64dbg 里的操作

在 CPU 窗口的某行指令上按 **X** 键，会弹出交叉引用列表窗口，显示所有引用这个位置的地方。

<!-- 📸 截图：按 X 键弹出的交叉引用列表 -->

**实际用法**：

- 在 "Correct!" 字符串引用处按 X → 看到哪些代码路径会走到这里
- 在一个函数的第一条指令按 X → 看谁调用了这个函数 → 追踪调用链
- 在一个全局变量地址按 X → 看哪些代码读取或修改了这个变量

这比搜索字符串更精确——搜索可能找到多处匹配，交叉引用只列真正有代码关联的位置。

### 交叉引用 vs 搜索字符串

| 方法       | 优点                       | 缺点                                   |
| ---------- | -------------------------- | -------------------------------------- |
| 搜索字符串 | 快速、直观、不依赖交叉引用 | 可能找到多处匹配，需要判断哪个是关键的 |
| 交叉引用   | 精确、只列真正的引用关系   | 需要先定位到目标位置才能用             |

实际逆向中经常交叉使用：先搜索字符串定位大致位置，再用交叉引用追踪调用链。

## 综合实战：用三种方法定位关键代码

场景：你拿到一个陌生的程序，想找到它的注册验证逻辑。

### 方法一：搜索字符串

1. 右键 → Search for → All referenced text strings
2. 找到 "注册成功" / "注册失败" / "Invalid serial" 之类的字符串
3. 双击跳转到引用它的代码
4. 在那附近找 `cmp`/`test` + `jne`/`je` 模式

这是最快的方法，大部分简单程序都能秒杀。

### 方法二：交叉引用追踪

1. 如果你知道验证函数的名字（比如 `check_serial`）
2. 在函数开头按 X → 看谁调用了它
3. 跳到调用方，看它传了什么参数
4. 再往上追踪，直到找到用户输入的处理逻辑

这适合程序结构比较清晰的情况。

### 方法三：内存窗口 + 断点

1. 搜索字符串找到注册码（可能是明文存储）
2. 在内存窗口定位注册码的地址
3. 在那个地址设**硬件断点**（右键 → Breakpoint → Hardware, on Access）
4. 运行程序，输入注册码
5. 程序会在读取这个字符串时断下来——你直接看到了比较的代码

这个方法最强大，即使字符串被加密也能用（运行时解密后在内存里是明文）。

### 实操练习：综合运用

打开第1章的 CrackMe 程序，依次尝试：

1. **搜索字符串** → 找到 "Correct!" → 双击跳转 → 看到验证代码
2. **在验证代码处按 X** → 看到跳转来源
3. **内存窗口 Ctrl+G** → 输入 "reverse2026" 的地址 → 看到明文密码
4. **修改内存中的密码** → 把 "reverse2026" 改成 "abcdefghij" → 运行输入新密码 → 看结果

## 练习

### 第一题

ASCII 字符串 "Hi" 的十六进制表示是什么？末尾的 `00` 是干什么用的？

<details>
<summary>答案</summary>

`48 69 00`（`H` = 0x48，`i` = 0x69，然后 `00` 是 NULL 终止符）。`00` 告诉程序"字符串到这里结束"——函数读取字符串时遇到 `00` 就停止。

</details>

### 第二题

UTF-16LE 字符串 "Hi" 的十六进制表示是什么？

<details>
<summary>答案</summary>

`48 00 69 00 00 00`。每个字符占 2 字节：`H` = `48 00`，`i` = `69 00`，末尾终止符 = `00 00`。

</details>

### 第三题

为什么程序传字符串时传的是地址而不是内容本身？

<details>
<summary>答案</summary>

因为字符串可能很长（几百甚至几千字节），寄存器只有 4 字节放不下。而且字符串的实际内容在内存里，传地址只需要 4 字节。函数拿到地址后自己去内存读内容。

</details>

### 第四题

x64dbg 实操：加载 CrackMe，在字符串列表中找到 "reverse2026"。双击跳转后，你在附近看到了什么指令？有没有看到 `push`、`call`、`test`、`jne` 这些我们已经学过的指令？

<details>
<summary>答案</summary>

你会看到类似这样的结构：

```asm
push <..."reverse2026"...>       ; push 字符串地址
...                               ; push 另一个参数
call <strcmp>                     ; 调用字符串比较函数
test eax, eax                    ; 检查返回值
jne <Wrong分支>                  ; 不相等则跳
push <..."Correct!"...>          ; 相等，显示成功
```

这些指令在前面几章都学过了。现在你应该能完全读懂这段代码。

</details>

### 第五题

x64dbg 实操：在 "Correct!" 字符串引用处按 X 键，看到了几条交叉引用？它们分别是什么含义？

<details>
<summary>答案</summary>

通常只有一条引用——就是验证成功后 `push` 这个字符串地址的指令。如果有多条，说明程序在多个地方输出了 "Correct!"，可能有多条验证路径。

</details>

### 第六题

x64dbg 实操：在内存窗口找到 "reverse2026" 字符串，用右键 → Binary → Edit 把它改成 "abcdefghij"（10 个字符，和原来一样长）。然后运行程序，输入 "abcdefghij"，程序告诉你什么？

<details>
<summary>答案</summary>

程序会显示 "Correct!"。因为你把内存中的正确密码从 "reverse2026" 改成了 "abcdefghij"，`strcmp` 比较输入和新密码相等，返回 0，走正确分支。

这就是**内存补丁**——不修改代码，直接改数据来绕过验证。

</details>
